# AXIOMS

## AX-MOD-API-001: Mongo client is a singleton tagged for Atlas attribution
The `api/` package must obtain its `MongoClient` from `api/lib/mongodb.ts`, which connects exactly once per Lambda cold start through `Resource.MONGODB_URI` and passes `appName: "brianstoker-api"`. Handlers must not instantiate their own `MongoClient` or strip the `appName` tag.

### Applies To
- `api/lib/mongodb.ts`
- `api/subscribe.ts`
- `api/sms.ts`
- any new handler added under `api/`

### Acceptance Checks
- `rg -n "new MongoClient" api/` returns only the single instantiation inside `api/lib/mongodb.ts`.
- `rg -n "appName" api/lib/mongodb.ts` shows `appName: "brianstoker-api"` is still present.
- manual: cold-start a deployed handler and confirm the Atlas connection list shows the `brianstoker-api` app name.

---

## AX-MOD-API-002: Handlers must validate `ROOT_DOMAIN` and origin before doing work
Every HTTP handler exported from `api/` must (a) reject the request with `400` when `process.env.ROOT_DOMAIN` is unset and (b) reject with `403` when `event.headers.origin` (with `https://` stripped) is not contained in `ROOT_DOMAIN`. These guards run before any I/O, Mongo access, SES send, or SNS publish.

### Applies To
- `api/subscribe.ts`
- `api/sms.ts`
- any new handler added under `api/`

### Acceptance Checks
- `rg -n "ROOT_DOMAIN" api/` shows the env-presence check and the origin allowlist check in every handler file.
- manual: invoke each handler with a missing `Origin` header or a foreign origin and confirm `403`.

---

## AX-MOD-API-003: Public handler signatures and response envelopes are a contract
The exported names and return shapes of `subscribe`, `verify`, and `handler` (SMS) are consumed by `stacks/api.ts` route bindings (currently commented but version-controlled). Renaming an export, changing it to a non-Lambda signature, or returning anything other than an `APIGatewayProxyResult`-compatible `{statusCode, headers?, body}` object is a breaking change that requires a governed task updating `stacks/api.ts` in the same change.

### Applies To
- `api/subscribe.ts` (exports `subscribe`, `verify`)
- `api/sms.ts` (exports `handler`)
- `stacks/api.ts`

### Acceptance Checks
- `rg -n "api/subscribe|api/sms|api\\.(subscribe|verify|handler)" stacks/` shows every reference still resolves to an export that exists in `api/`.
- `pnpm typescript` passes.

## AX-MOD-API-004: Subscriber redirect codes are observable behavior
The `verify(event)` handler in `api/subscribe.ts` issues `301` redirects to `https://${ROOT_DOMAIN}/subscription?code=<N>&email=...`, where `<N>` is one of `200|201|401|402|404|500`. Each numeric code corresponds to a documented branch (success, already-verified, bad/missing token, missing DB config, unknown email, internal error). Adding, removing, or repurposing a code requires updating the consuming `/subscription` view in the Next.js app at the same time.

### Applies To
- `api/subscribe.ts` (`verify` function, lines ~141-223)
- the `/subscription` page in the Next.js app once it exists/changes

### Acceptance Checks
- `rg -n "code=(200|201|401|402|404|500)" api/subscribe.ts` lists exactly the documented set.
- manual: any change to the code set is accompanied by a matching change to the `/subscription` page or a governed migration ticket.

## AX-MOD-API-005: SMS payloads are restricted to validated E.164 phone numbers
`sms.handler` must reject any request whose `phoneNumber` does not match `/^\+\d{11,15}$/` with HTTP `400` before constructing an SNS `PublishCommand`. This guard exists to prevent accidental spend on malformed destinations.

### Applies To
- `api/sms.ts`

### Acceptance Checks
- `rg -n "\\^\\\\+\\\\d\\{11,15\\}\\$" api/sms.ts` finds the regex.
- manual: POST with `phoneNumber: "1234"` returns `400` and no SNS call is made.

---

## AX-MOD-API-006: Outbound AWS service clients are pinned to `us-east-1` and email originates from `no-reply@${ROOT_DOMAIN}`
Both AWS service clients in this package (`SES` in `api/subscribe.ts`, `SNSClient` in `api/sms.ts`) must be constructed with `region: "us-east-1"`, and `sendVerificationEmail()` must send with `Source: no-reply@${process.env.ROOT_DOMAIN}`. This pairs the senders with the region where the SES domain identity (account `167217327520`) and SNS SMS configuration live; changing the region or the `Source` address without a matching SES/SNS identity migration silently breaks email/SMS delivery and the (commented) `ses:SendEmail` IAM ARN in `stacks/api.ts`.

### Applies To
- `api/subscribe.ts` (`ses` client + `sendVerificationEmail`)
- `api/sms.ts` (`snsClient`)
- `stacks/api.ts` (the `arn:aws:ses:us-east-1:167217327520:identity/*` grant)

### Acceptance Checks
- `rg -n 'region: "us-east-1"' api/subscribe.ts api/sms.ts` shows the region pin in both clients.
- `rg -n 'no-reply@\$\{process.env.ROOT_DOMAIN\}' api/subscribe.ts` shows the verification `Source`.
- manual: any region or `Source` change is accompanied by a matching SES identity / SNS config update and the `stacks/api.ts` IAM ARN.
