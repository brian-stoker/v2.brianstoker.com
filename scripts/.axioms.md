# AXIOMS

## AX-MOD-SCRIPTS-001: `local-sync-cron.cjs` must stay alive when `SYNC_SECRET` is absent
The local sync cron driver must NOT exit when `process.env.SYNC_SECRET` is missing. It must log a warning and remain running (currently via `setInterval(() => {}, 1 << 30)`) so that `turbo watch dev:cron` does not treat the process as a crashed task. It must also ignore `SIGHUP` for the same reason. The bearer-token contract with `/api/github/sync-events` is `Authorization: Bearer ${SYNC_SECRET}` over `POST`, with an optional `?fullRefresh=true` query parameter driven by `FULL_REFRESH=true`.

### Applies To
- `scripts/local-sync-cron.cjs`

### Acceptance Checks
- `rg -n "SYNC_SECRET" scripts/local-sync-cron.cjs` shows the env-presence check, the warning log, and the keep-alive `setInterval`.
- `rg -n "SIGHUP" scripts/local-sync-cron.cjs` shows the ignored-SIGHUP handler.
- `rg -n "Authorization.*Bearer.*SYNC_SECRET" scripts/local-sync-cron.cjs` finds the bearer-token header construction.
- manual: run `SYNC_SECRET= pnpm dev:cron` and confirm the process stays alive without spinning.

## AX-MOD-SCRIPTS-003: Production deploys go through `aws-deploy.sh`, repoint CloudFront, then ship logs
Production deployment must be invoked via `scripts/aws-deploy.sh deploy` (i.e. `pnpm deploy:prod`). On `deploy` the script must run three steps in order: (a) `senvn -f production npx sst deploy --stage production`, (b) `AWS_PROFILE=stokd-cloud node scripts/update-cloudfront-origins.cjs` (repoint origins off the broken Lambda Function URLs â€” required or the site 403s), (c) `AWS_PROFILE=stokd-cloud node scripts/setupLogShipping.cjs`, treating only the log-shipping failure as non-fatal (warning printed, overall exit 0). The script must accept only `deploy` and `remove` as actions and reject any other argument. AWS account verification is intentionally delegated to `senvn` + `.deploy.json`; do not reintroduce ad-hoc `aws sts` checks in this script. (Refresh note: steps b and the `stokd-cloud` profile were added/changed during the `stoked â†’ stokd-cloud` account migration; the prior axiom text referenced `AWS_PROFILE=stoked` and only two steps.)

### Applies To
- `scripts/aws-deploy.sh`
- `scripts/update-cloudfront-origins.cjs`
- `scripts/setupLogShipping.cjs`

### Acceptance Checks
- `rg -n "senvn -f production npx sst (deploy|remove) --stage production" scripts/aws-deploy.sh` finds both invocations.
- `rg -n "AWS_PROFILE=stokd-cloud node scripts/update-cloudfront-origins.cjs" scripts/aws-deploy.sh` is present (step b).
- `rg -n "AWS_PROFILE=stokd-cloud node scripts/setupLogShipping.cjs" scripts/aws-deploy.sh` is present and followed by `|| echo` (step c, non-fatal).
- `rg -n "set -euo pipefail" scripts/aws-deploy.sh` is present.
- manual: run `./scripts/aws-deploy.sh foo` and confirm it prints the usage line and exits 1.

## AX-MOD-SCRIPTS-007: `update-cloudfront-origins.cjs` is the idempotent post-deploy reconvergence of CloudFront onto API Gateway
Because Lambda Function URLs return 403 in AWS account `167217327520`, the production site is served by API Gateway HTTP APIs that proxy the SST-built Lambdas, and SST does not manage those APIs or the CloudFront origin overrides. `update-cloudfront-origins.cjs` is the only sanctioned reconverger: it must remain self-contained and idempotent, must operate on distribution `E1JN9JWBQ37JT2` and exactly the two origins `default` / `imageOptimizer`, must discover the Lambdas by the `brianstokdcloudNextjsSiteDefaultFunction` / `brianstokdcloudNextjsSiteImageOptimizerFunction` name patterns, must (re)create one `AWS_PROXY` HTTP API per Lambda, and must ensure the `apigateway` invoke permission via a **remove-then-add** where only the `remove-permission` may fail (a swallowed `add-permission` leaves the Lambda un-invokable and every request 500s). Each origin must be set to an https-only custom origin with the OAC cleared, followed by a `/*` invalidation. It runs after every `sst deploy` (step b of `aws-deploy.sh`).

### Applies To
- `scripts/update-cloudfront-origins.cjs`

### Acceptance Checks
- `rg -n "E1JN9JWBQ37JT2" scripts/update-cloudfront-origins.cjs` shows the pinned distribution id.
- `rg -n "default:|imageOptimizer:|brianstokdcloudNextjsSite" scripts/update-cloudfront-origins.cjs` shows the two origins and the Lambda name patterns.
- `rg -n "remove-permission|add-permission|awsAllow" scripts/update-cloudfront-origins.cjs` shows the remove-then-add idempotency (remove tolerated, add must succeed).
- `rg -n "https-only|OriginAccessControlId" scripts/update-cloudfront-origins.cjs` shows the https-only custom origin with OAC cleared.
- manual: after `pnpm deploy:prod`, `https://brian.stokd.cloud` returns 200 (not 403) once the invalidation propagates.
