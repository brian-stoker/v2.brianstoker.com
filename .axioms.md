# AXIOMS

## AX-MOD-V2BS-001: GitHub data is read from MongoDB, never fetched in the page-request path
The `/work` activity feed and its read APIs must serve GitHub events exclusively
from MongoDB. The GitHub Events API is contacted only by the sync path
(`lib/github-sync.ts` via the hourly cron or the secret-gated trigger), never
synchronously inside a page render or a public GET handler. This keeps page loads
off GitHub's rate limit and resilient to GitHub outages.

### Applies To
- `pages/api/github/events.ts`
- `pages/api/github/filters.ts`
- `pages/work.tsx`
- `lib/github-sync.ts`
- `cron/github-sync.ts`

### Acceptance Checks
- `manual: confirm pages/api/github/events.ts and filters.ts read only via getDatabase()/Mongo and contain no api.github.com fetch in the request path.`
- `grep -RIlF "api.github.com" pages/api/github/events.ts pages/api/github/filters.ts` returns no matches (use `-F`: a regex dot otherwise false-matches `/api/github/com…` paths).

## AX-MOD-V2BS-002: The GitHub sync trigger requires the shared SYNC_SECRET bearer
`POST /api/github/sync-events` must reject non-POST with **405** and any request
whose `Authorization` header is not `Bearer ${SYNC_SECRET}` with **401**, before
running `syncGitHubEvents()`. The same `SYNC_SECRET` value is the shared contract
across the EventBridge cron, `scripts/local-sync-cron.cjs`, and this endpoint;
rotating one side without the others breaks the hourly and local sync flows.

### Applies To
- `pages/api/github/sync-events.ts`
- `scripts/local-sync-cron.cjs`
- `stacks/cron.ts`

### Acceptance Checks
- `grep -nE "status\(401\)|status\(405\)|Bearer \\$\\{process.env.SYNC_SECRET\\}" pages/api/github/sync-events.ts` shows all three guards present.
- `manual: a POST without the correct bearer returns 401 (not 500) and does not invoke syncGitHubEvents.`

## AX-MOD-V2BS-003: All MongoDB access goes through the memoized getDatabase singleton
Handlers and the sync logic must obtain the database via the memoized
`getDatabase()` client (HMR-safe, single `MongoClient` promise) rather than
constructing their own connections. The three near-duplicate connection modules
(`pages/api/lib/mongodb.ts`, `lib/mongodb.ts`, `api/lib/mongodb.ts`) must stay
consistent; a fix in one generally requires the same fix in the others. The DB
name must continue to derive from `MONGODB_NAME`
(`brianstoker-production` / `brianstoker-local`).

### Applies To
- `pages/api/lib/mongodb.ts`
- `lib/mongodb.ts`
- `api/lib/mongodb.ts`
- `lib/github-sync.ts`

### Acceptance Checks
- `grep -nE "getDatabase|MongoClient" pages/api/lib/mongodb.ts lib/mongodb.ts` confirms a single memoized client per module.
- `manual: no handler under pages/api/** instantiates a bare new MongoClient(...) outside the lib modules.`

## AX-MOD-V2BS-004: Changing the events API response shape requires bumping the client CACHE_VERSION
The Work view caches `/api/github/events` results in `localStorage` keyed by
`CACHE_VERSION` (currently `'5.0'`) in `src/utils/eventCacheManager.ts`. Any change
to the events API response shape, projection, or the pruned index-entry shape
(`src/utils/eventCachePruning.ts`) must bump `CACHE_VERSION` so existing clients
discard stale caches instead of rendering malformed data.

### Applies To
- `src/utils/eventCacheManager.ts`
- `src/utils/eventCachePruning.ts`
- `pages/api/github/events.ts`

### Acceptance Checks
- `grep -n "CACHE_VERSION" src/utils/eventCacheManager.ts` shows the version constant.
- `manual: any PR that alters the events response/projection also bumps CACHE_VERSION in eventCacheManager.ts.`

## AX-MOD-V2BS-005: Required environment variables are validated at synth and must not be silently renamed
`stacks/site.ts` aborts SST synth when a required env var is missing
(`ROOT_DOMAIN`, `MONGODB_*`, `GITHUB_TOKEN`/`GITHUB_USERNAME`, `SYNC_SECRET`,
`GOOGLE_CLIENT_*`, `NEXTAUTH_*`, `S3_BUCKET_NAME`, …). These names are an external
contract spanning `.env*`, GitHub Actions secrets, and `stacks/*`. Renaming or
removing one requires a coordinated migration across all three, not a local edit.

### Applies To
- `stacks/site.ts`
- `stacks/envVars.ts`
- `sst.config.ts`
- `.github/workflows/deploy-production.yml`

### Acceptance Checks
- `manual: required env var names in stacks/site.ts match .env.production keys and the CI workflow secret list.`
- `pnpm typescript` succeeds (no broken env references after a rename).

## AX-MOD-V2BS-006: `/hal` is the only authenticated surface; all other views remain anonymous
The HAL log viewer (`pages/hal.js` + `pages/api/hal/logs.js`) is the single
auth-gated surface, protected by the NextAuth Google OAuth allow-list
(`brianstoker@gmail.com`, `brian@stokd.cloud`) in
`pages/api/auth/[...nextauth].js`. No other public route may require
authentication, and the allow-list must not be widened without a governed task.

### Applies To
- `pages/hal.js`
- `pages/api/hal/logs.js`
- `pages/api/auth/[...nextauth].js`

### Acceptance Checks
- `grep -nE "brianstoker@gmail.com|brian@stokd.cloud" pages/api/auth/[...nextauth].js` confirms the 2-email allow-list.
- `manual: no page outside /hal calls useSession()/signIn() as an access gate.`

## AX-MOD-V2BS-007: Existing tests and guard messages are observable behavior and must be preserved
The Playwright suite (`e2e/**`, 7 specs / 9 viewport projects, plus committed
visual baselines) must stay green before any mutation merges. The guard error
strings thrown by `syncGitHubEvents()` — `'GitHub token not configured'` and
`'Database not available'` — and the `maxPages = 7` page cap are observable
behavior referenced by docs/`SC_TEST.md`; rewording or removing them requires
updating those references in the same change. Per the TDD axiom, behavioral
changes add a failing test first.

### Applies To
- `lib/github-sync.ts`
- `e2e/**`
- `playwright.config.ts`

### Acceptance Checks
- `grep -nE "GitHub token not configured|Database not available|maxPages = 7" lib/github-sync.ts` shows the guards intact.
- `pnpm test:smoke` (and `pnpm test:visual` on intentional UI changes) passes.

## AX-MOD-V2BS-008B: CalendarBooking component UI/UX invariants (TDD: tests first)
The CalendarBooking component follows strict TDD discipline: acceptance tests
are written first and intentionally fail (RED), then the component is built
to make them pass (GREEN). The UI must match the reference design: calendar
month picker on left, week view with 7 days + time slots on right, clicking
a slot selects it and shows the form, duration is adjustable (30–120 min,
15 min increments) by dragging top/bottom edges. Form submission validates
name/email/phone (required) and company/reason (optional), then POSTs to
`/api/calendar/book`. Availability is fetched from `/api/calendar/availability`
keyed by date. Dark/light mode supported via MUI theme.

### Applies To
- `e2e/book-time.spec.ts` (acceptance tests, written first)
- `src/components/CalendarBooking/CalendarBooking.tsx` (implementation, written after tests)
- `/pages/book-time.tsx` (integration)

### Acceptance Checks
- `pnpm test:smoke e2e/book-time.spec.ts` runs and **fails** (RED phase) before implementation
- Tests cover: date selection, API fetching, slot clicking, form appearance, validation, duration adjust, submission
- After implementation, same test suite passes (GREEN phase)
- UI matches reference: calendar left + week right, clickable slots, form appears on selection, drag handles visible

## AX-MOD-V2BS-008: Google Calendar API integration goes through service account in pages/api/calendar
The appointment booking surface (`/book-time` page + `CalendarBooking` component)
integrates with Google Calendar API (calendar ID `brian@stokd.cloud`) via two
endpoints: `GET /api/calendar/availability` (fetch free slots) and
`POST /api/calendar/book` (create events with form data). Both must use a shared
service account key stored in `GOOGLE_CALENDAR_SERVICE_ACCOUNT` env var
(JSON string). Form data (name, email, phone, company, reason) is embedded in
the event description/notes only; no separate persistence. Both endpoints validate
required fields and reject with **400** (bad input), **401** (missing env), or **500** (API error).

### Applies To
- `pages/api/calendar/availability.ts`
- `pages/api/calendar/book.ts`
- `stacks/site.ts`
- `src/components/CalendarBooking/` (in sui-common, imports from this repo's API)

### Acceptance Checks
- `grep -n "GOOGLE_CALENDAR_SERVICE_ACCOUNT" stacks/site.ts` shows env var validation at synth.
- `grep -nE "status\(400\)|status\(401\)|status\(500\)" pages/api/calendar/{availability,book}.ts` shows all error paths guarded.
- `manual: GET /api/calendar/availability?date=2026-06-15 returns available 30min slots; POST /api/calendar/book with form data creates event in brian@stokd.cloud.`
